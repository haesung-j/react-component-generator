# TDD Rule

> **이 규칙은 Rigid — 상황에 맞게 변형하지 마라.**

## 프로젝트별 규칙 우선 조항

이 파일은 **기본값(fallback)** 이다. 현재 프로젝트의 `CLAUDE.md` 또는 `AGENTS.md`에 TDD 관련 규칙이 명시되어 있으면 **그 규칙을 우선 적용**하고 이 파일은 무시한다.

---

## 1. 적용 기준

### TDD를 반드시 적용해야 하는 대상

- **비즈니스 로직** — 조건 분기, 계산, 상태 전이
- **API 엔드포인트** — 요청/응답 계약, 에러 처리
- **유틸리티 함수** — 순수 함수, 데이터 변환, 포맷터
- **버그 수정** — 재현 테스트를 먼저 작성한 뒤 수정

### TDD가 불필요한 대상

- **타입 정의** — `interface`, `type`, `enum` 선언
- **설정 파일** — `vite.config.ts`, `eslint.config.js`, `.env`
- **순수 UI** — 레이아웃, 색상, 애니메이션 등 시각적 렌더링만 담당하는 컴포넌트
- **SQL / 스키마 마이그레이션** — DDL, 마이그레이션 파일

---

## 2. RED-GREEN-REFACTOR 사이클

### RED — 실패하는 테스트 작성

1. **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
2. **반드시 실행해서 실패를 눈으로 확인한다.** "당연히 실패하겠지"로 넘어가지 마라.
3. **실패 이유가 "기능 미구현"이어야 한다.** 컴파일 에러나 잘못된 테스트 설계로 인한 실패는 RED가 아니다 — 먼저 고쳐라.

### GREEN — 최소한의 코드로 통과

1. **테스트를 통과시키는 가장 단순한 코드만 작성한다.**
2. **YAGNI.** 지금 테스트가 요구하지 않는 기능은 구현하지 않는다.
3. **신규 테스트와 기존 테스트가 모두 통과하는지 확인한다.**

### REFACTOR — 동작 유지, 설계 개선

1. **중복 제거, 이름 개선, 헬퍼 추출**만 허용한다.
2. **모든 테스트가 green 상태를 유지해야 한다.** 리팩터 중 테스트가 깨지면 즉시 되돌린다.
3. **새로운 동작을 추가하지 마라.** 새 동작이 필요하면 리팩터를 마친 후 다시 RED로 돌아간다.

### 반복

리팩터가 끝나면 **다음 동작에 대한 RED**로 돌아간다.

---

## 3. 삭제 강제 규칙

- 테스트보다 프로덕션 코드를 먼저 작성했다면 **프로덕션 코드를 삭제하고 RED부터 재시작한다.**
- **"참고용으로 남겨두는 것"도 금지한다.** 주석 처리도 허용하지 않는다.

---

## 4. 변명 차단표

| 변명 | 반론 |
|------|------|
| "너무 단순해서 테스트가 불필요하다" | 단순한 코드가 가장 먼저 망가진다. 단순할수록 테스트 비용이 낮다. |
| "나중에 추가하겠다" | 나중은 오지 않는다. 코드가 복잡해지면 테스트 추가 비용은 10배가 된다. |
| "시간이 없다" | TDD 없이 작성한 코드의 디버깅 시간이 더 길다. 지금 쓰는 시간이 더 짧다. |
| "삭제하면 낭비다" | 테스트 없는 코드를 남기는 것이 낭비다. 잘못된 방향의 코드를 빨리 쓰는 것도 낭비다. |
| "프로토타입이다" | 프로토타입 코드는 프로덕션에 들어가지 않을 때만 면제된다. 프로덕션에 들어갈 거라면 프로토타입이 아니다. |
